Authentication

All requests must be authenticated using a Google Account and the OAuth protocol. The access token can be requested as described here. The access token must be transmitted as secret using HTTP BASIC authentication. All requests must be done over SSL.

HTTP methods usage

GET

Retrive the resource representation.

PUT

Send the intended resource representation in the request body. Modifies a resource or creates it if it doesn't exist.

POST

Send data to be processed to a resource, using x-www-form-urlencoded MIME type.

DELETE

Delete the resource.

HTTP Status codes

Each request may return one of these status codes:

200

The request has been completed successfully.

201

The request has been completed successfully. A new resource has been created.

204

The request has been completed successfully. The response doesn't contain any entity.

400

Bad representation.

404

The resource doesn't exist.

405

The HTTP method is not handled for this resource.

412

The server failed to process the request due to wrong or missing previous related requests.

422

The request was well-formed but was unable to be followed due to semantic errors,

500

There was an error during the request handling,



Resource description

/group

DESCRIPTION

A group is a data-collector intended to gather tables with the same specification.

URL Stucture

https://kd-cloud.appspot.com/group/<name>

name the unique name of the group

METHODS

PUT/GET

REPRESENTATION

XML

Specifies the data specification to be accepted. See data specification.

Example

<dataSpecification>

<column>

<name>heartbeat</name>

<type>DOUBLE</type>

<source>HEARTBEAT</source>

</column>

</dataSpecification>

/data

DESCRIPTION

A group subresource. The actual data collected by the group. Data is then divided by user.

URL Stucture

https://kd-cloud.appspot.com/group/<group>/data

group the unique name of the group

METHOD

PUT/GET

REPRESENTATION

A table of data encoded according to the HTTP Content-Type

CSV (with header record) for text/csv

Arff for text/plain

Json for application/json

XML specifications

All XML inputs can be validated using the following schema.

Modalities

structure

<modality>

<name/>

<inputSpecification/>

<init-action/>

<action/>

<outputSpecification/>

</modality>

tags

  • name is a generic name for the modality

  • inputSpecification/outputSpecification are data specifications, each one is needed if the modality respectively sends or receives data

  • init-action/action are actions, init-action is optional

Data specification

Structure

<dataSpecification>

<column>

<name/>

<type/>

<source/>

</column>

<view/>

</dataSpecification>

tags

column

A table column. Multiple columns can be specified

name

The name of the column

type

The datatype of the column. Supported types are DOUBLE, INTEGER, TIMESTAMP

source

Optional. Specifies the source sensor to use to gather data. Currently supported ones are HEARTBEAT, CLOCK

view

Optional. Specify the relative URI of the view specification

Workflows

Structure

<workflow>

<node>

<type/>

<parameter>

<name/>

<value/>

</parameter>

</node>

</workflow>

tags

node

An execution unit. Additional nodes can be uploaded as plugins.

type

The node type which identifies what the node actually does. Basically it's a Java class implementing the Node interface

parameter

A static parameter of a node. Multiple workflows could use the same node with different parameters

name

The name of the parameter

value

The value of the parameter


Actions

Structure

<action>

<uri/>

<method/>

<parameter>

<name/>

<value/>

<reference/>

<parameter/>

<repeat/>

<trigger/>

</action>

tags

uri

The relative URI to use for the request

method

The HTTP method to use for requests

parameter

Only valid for POST requests. Specifies a parameter for POST forms

name

The name of the parameter

value

Optional. The value of the parameter

reference

Optional. The value of the parameter will be taken using an xpath expression. The expression is specified using the xpath attribute. An attribute type defines the reference type, supported types are CHOICE (default) or MAP. See modality execution for details.

repeat

Optional. Can be either true or false. Default false. Specifies if the action must be repeated

trigger

Optional. Specifies (through XML attributes) a trigger that activates the action. Currently the only trigger supported is after, which activates the action after the specified time (in seconds)

Notes

  • All PUT bodies must be an encoded table

  • All POST bodies must be an post form (application/x-www-form-urlencoded)

  • A parameter must include either a value or a reference

Modality Execution

  • The client must start collecting data according to inputSpecification

  • The init-action is intended to either prepare the server for the proper action or to retrieve runtime informations. The (eventual) XML output of the action must be stored. The init-action can be executed at the beginning or when triggered if a trigger is specified. Eventual parameters in this action cannot require references

  • The action must be executed after the init-action (if any) and when triggered (if a trigger is specified). Parameter references should be resolved as follows:

    • use the init-action output XML to evaluate the xpath expression

    • if the reference is of type CHOICE (default) then let the user choose between the expression results

    • If the reference is of type MAP a different request must be made using each of the expression results

    The action output must match the outputSpecification (if it is present, otherwise must be empty). The client must also fetch the view specification according to the URI enclosed in the view tag and generate the report.

Workflow nodes and execution

A workflow describes a set of operations the server engine should perform. Unlike the traditional workflows, kdcloud ones are merely a sequence of jobs (nodes) following each other, the output of the previous node is given as input to the next one, and the final output (if any) is returned to the user.

A node may have static and dynamic parameters. Static parameters are defined directly in the workflow XML, dynamic parameters must be supplied in the HTTP POST body.

Additional nodes can be created implementing the Node interface, instructions on how develop a node can be found here. Once uploaded, they can be used in workflows. A node should only perform operations on data, but the platform also provide two built-in node for accessing the datastore:

  • UserDataReader. Reads data given a user and a group.

    Dynamic Parameters

    • sourceGroup

    • sourceUser

  • UserDataWriter. Writes data to a group by username.

    Dynamic Parameters

    • destinationGroup

    • destinationUser
